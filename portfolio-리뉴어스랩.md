# 리뉴어스랩 — VoC 자동 분류·인사이트 파이프라인

**김민수** — 리뉴어스랩 CS팀의 VoC 수작업 정리, **210분 → 20분(약 90% 단축)** 으로 줄인 자동 분류·인사이트 파이프라인

---

## 1. 한 장 요약 (Executive Summary)

- **받은 문제**: "고객사로부터 수집된 VoC CSV 데이터를 분석하여, 유형별(불만·기능 요청·칭찬·일반 문의) 분류표를 만들고 탄소규제(CSRD·CBAM) 맥락에서 비즈니스 의미 있는 인사이트 3개를 도출하라." (리뉴어스랩 미션 원문)
- **만든 것**: CSV 한 장만 넣으면 정제 → 4유형·15소분류·5원인태그 자동 분류 → 규제 캘린더 대조 인사이트 3개 → 우선순위 점수 기반 제품 개선 기획안 2개 → 구글시트 자동 기입까지 사람 개입 없이 끝나는 파이프라인.
- **핵심 임팩트**: VoC 월간 처리 시간 **210분 → 20분** (추정치, 산출 근거는 5번 섹션). 44건 중 긴급 대응 필요 건 2건을 자동 감지.
- **데모**:
  - 분류 리포트: `output/report_voc.html`
  - 구글시트 연동 결과: [구글스프레드시트 스크린샷 첨부]

## 2. 문제 이해

**기업 제출 원문 문제**
> "탄소규제 컨설팅 특성상 고객 VoC는 이메일, 채널톡, 설문, 미팅 등 다양한 채널로 산발적으로 들어옵니다. 특히 CBAM 확정 시행(2026년 1월), CSRD 적용 확대 등 규제 변화가 맞물리면서 문의와 불만이 급증하고 있습니다. 현재 CS팀은 VoC를 수작업으로 엑셀에 정리하고 있어 유형별 패턴 파악이 어렵고, 비즈니스 의사결정에 활용하지 못하고 있습니다."

**내가 재정의한 문제**
단순히 "VoC를 4가지로 나누면 되는 문제"가 아니라, **매달 반복해서 CS팀장·제품 PM에게 "지금 뭘 해야 하는가"를 전달하는 문제**로 다시 정의했다. 분류표만 만들면 다음 달 새 CSV가 들어왔을 때 또 사람이 처음부터 다시 읽어야 한다. 그래서 "이번 달만 잘 분류하는 것"이 아니라 "매달 CSV만 바꿔 끼우면 동일 품질의 판단 자료가 나오는 구조"를 목표로 잡았다.

**문제의 크기 (Before, 추정치)**
- 월평균 약 30건(수집 기간 45건 ÷ 1.5개월)을 사람이 하나씩 읽고, 유형 판단하고, 엑셀에 입력 → 건당 5분 × 30건 = 150분
- 월말 건수·비율 집계 20분 + 규제 캘린더와 대조해 인사이트 도출 40분 = 60분
- **합계 210분/월(3.5시간)** — CS팀장 1명이 한 달에 반나절을 VoC 정리에만 쓰는 구조

**스코프**
- **이번에 푼 것**: CSV 정제, 4유형+15소분류+5원인태그 분류, 긴급 건 자동 플래그, 규제 캘린더 연동 인사이트 3개, 우선순위 점수 기반 기획안 자동 생성, 구글시트 자동 기입
- **의도적으로 안 푼 것**: 이메일·채널톡·미팅메모를 자동으로 한 CSV로 통합하는 것(제약조건상 voc.csv만 사용), 실시간 처리, 월별 트렌드 자동 비교(현재는 단일 기간 스냅샷 분석)

## 3. 해결 접근 & 의사결정

**접근 가설**
"이번 달 44건을 잘 분류하는 것"보다 "다음 달, 그다음 달에도 같은 품질이 나오는 것"이 CS팀·제품팀에게 더 큰 가치라고 판단했다. 그래서 분류 기준을 코드가 아니라 문서(`decisions.md`)에 먼저 고정하고, 코드는 그 문서를 그대로 따라가게 만들었다.

**버린 대안 + 버린 이유**
- 처음에는 기획안 선정을 "불만 건수 1위 + 기능요청 건수 1위"로 고정하려 했다. 그런데 실제로 급한 것(OEM 고객이 겪은 시스템 오류, 2건)보다 단순히 건수가 많은 요청(4건)이 기획안으로 뽑히는 역설이 생겼다. **건수만 보면 "많이 들어온 것"과 "정말 급한 것"이 다르다**는 걸 발견하고, 건수·OEM 고객 여부·긴급 여부·제품 결함 여부를 합산한 점수제로 바꿨다. 선정 기준과 채점 기준을 하나로 통일한 것이 핵심 결정이었다.
- 긴급 건 분류도 처음엔 "이 문의가 불만이냐 문의냐"만 보면 된다고 생각했다. 그런데 "CBAM 신고 마감을 놓친 문의"는 유형상 '일반 문의'이지만 대응 속도는 '불만'보다 급하다는 걸 발견해서, **유형 분류와 긴급도를 별도의 두 축으로 분리**했다.

**핵심 흐름** (사람이 알아볼 동작 기준)
```
CSV 업로드
  → 날짜·중복·결측 자동 정리
  → 문의 내용 읽고 4유형(불만/기능요청/칭찬/일반문의) + 세부 원인 분류
  → "지금 규제 일정(CBAM 마감 등)과 겹치는 부분이 있는지" 자동 대조
  → CS팀·제품팀이 봐야 할 인사이트 3개 + 개선 우선순위 2건 자동 작성
  → 구글시트에 색깔 입혀서 자동 기입 (긴급 건은 별도 탭)
```

**막혔던 지점 & 돌파 방법 (★가장 중요★)**
분류 로직 자체보다 더 걱정했던 건 "이걸 실제로 CS팀장이 매달 쓸 수 있을까"였다. 이 파이프라인을 만든 나는 터미널 사용이 익숙하지만, 실제 사용자는 코드를 모르는 비기술직이다. **아무리 정교하게 만들어도 실제 사용자가 실행조차 못 하면 무용지물**이라고 판단해서, 별도로 `HOW_TO_USE.md`라는 "복사→붙여넣기만 하면 되는" 실행 가이드를 만들었다.

실제로 테스트해보니 예상대로 문제가 터졌다. 프로젝트 폴더 이름에 `[01]`처럼 대괄호가 들어있으면, 윈도우 터미널(PowerShell)이 이걸 폴더 이름이 아니라 "특정 패턴을 찾으라"는 명령으로 잘못 해석해서 "경로를 찾을 수 없다"는 오류가 났다. 기술을 모르는 사용자 입장에선 이 오류 메시지만 보고는 뭘 고쳐야 할지 전혀 알 수 없는 상황이었다. 원인을 찾아 `Set-Location -LiteralPath` 라는, 폴더 이름을 있는 그대로 읽으라는 명령으로 바꿔 해결했고, 가이드 문서에 이 부분을 굵은 글씨로 강조해뒀다. 겉으로는 사소해 보이지만, 이 한 줄을 못 넘으면 CS팀장은 파이프라인을 아예 실행조차 못 한다는 점에서 가장 중요하게 다룬 문제였다.

**AI vs 나의 판단**
- **AI(Claude)에게 맡긴 것**: 44건 각각의 내용을 읽고 초기 유형·소분류를 판단하는 반복 작업, 규제 캘린더와 VoC 패턴을 대조해 인사이트 초안을 작성하는 작업
- **내가 직접 결정한 것**: 4유형의 판단 기준 자체(예: "기존 기능을 고쳐달라"=불만 vs "새 기능을 만들어달라"=기능요청), 기획안 선정을 점수제로 바꾼 결정, 긴급도와 유형을 분리하기로 한 결정, 사용자가 실제로 쓸 수 있는 형태(가이드 문서)로 만드는 것 — 즉 "무엇을 기준으로 나눌 것인가"와 "누가 실제로 쓸 것인가"는 전부 내 판단 영역으로 남겼다.

**실제 동작 1건** (2026-05-11 자 실제 VoC)
> 입력: *"카본링크에서 출력한 CBAM 보고서가 EU 세관에서 반려됐습니다. 양식 문제인지 수치 문제인지 확인 부탁드립니다. 긴급합니다."* (OEM 고객, 이메일)

**무엇을 하는지만 (기술 디테일은 별첨으로)**: 사람이 44건을 하나씩 읽으며 "이건 급한가, 아닌가"를 판단하던 일을, 이 파이프라인은 문의 내용 속의 표현(마감·반려·긴급 등)을 보고 자동으로 골라내 CS팀장 눈앞에 먼저 보여준다.

## 5. 임팩트

| 핵심 지표 | Before (수작업, 추정) | After (파이프라인) |
| --- | --- | --- |
| 월간 VoC 처리 시간 | 210분 (건당 분류·입력 150분 + 월말 집계·인사이트 도출 60분) | 약 20분 (CSV 교체 1분 + 실행 3분 + 리포트 검토 15분) |
| 긴급 건 파악 | 44건을 다 읽어야 확인 가능 | 자동 플래그로 즉시 2건 노출 |
| 인사이트 도출 | 수작업 재구성, 근거 비체계적 | 규제 캘린더 대조 기반 3개 자동 생성 |

**정성 효과**
- CS팀장이 "이번 달 뭐가 급한지"를 44건을 다 읽지 않고도 리포트 최상단에서 바로 파악 가능
- 제품 PM은 건수가 아니라 (OEM 포함 여부·긴급도·제품 결함 여부를 반영한) 우선순위 점수로 다음 스프린트 후보를 받는다
- 분류 기준과 경계 케이스 판단 근거가 문서로 누적되어, 다음 담당자가 바뀌어도 "왜 이렇게 분류했는지"를 그대로 이어받을 수 있다

**측정 방식 / 가정 명시**
- 위 시간 수치는 **CS팀 실측 타임로그가 아닌 추정치**다. `data/voc.csv` 건수와 컨설팅 업계 VoC 처리 실무를 근거로 "취합 1.5분 + 판단 2.5분 + 입력 1분"처럼 단계별로 역산했다. 과장을 피하기 위해 "내용을 읽고 이해하는 시간"은 자동화 이후에도 리포트 검토 15분으로 남겨뒀다. 실제 배포 시에는 CS팀 실측값으로 이 수치를 갱신해야 한다.

## 6. 적용 가능성 & 아쉬움·비전

**우리 환경 적용 시 필요한 것**
- 매달 CS팀이 이메일·채널톡·미팅메모 내용을 하나의 CSV로 정리하는 최소한의 습관 (채널 자동 통합은 이번 범위 밖)
- 구글 스프레드시트 서비스 계정 공유 설정(최초 1회)
- 규제 일정이 바뀔 때 `context/reg-calendar.md` 한 줄 업데이트(PM)

**아쉬웠던 부분 (정직하게)**
- 핵심 임팩트 수치(210분→20분)가 실측이 아닌 추정치다. CS팀 실제 작업 시간을 재보지 못한 채 3시간짜리 개인 과제 안에서 합리적으로 역산한 값이라 한계가 있다.
- 새로운 소분류나 새로운 긴급 표현이 등장하면 자동으로 학습되지 않고, 사람이 `decisions.md`와 `URGENT_KW` 목록을 수동으로 업데이트해야 한다. "완전 자동화"보다는 "사람의 판단을 문서로 누적시키는 반자동화"에 가깝다.
- 월별 데이터 간 트렌드 비교(예: 이번 달 불만 비율이 지난달 대비 늘었는지)는 아직 지원하지 않는다. 월 단위 스냅샷 분석까지만 되어 있다.

**다음 스텝 (2주 더 주어진다면)**
1. CS팀 실측 타임로그를 확보해 210분→20분 추정치를 실측값으로 교체
2. 월별 classified_voc CSV를 누적 저장해 "지난달 대비 이번 달" 트렌드 인사이트 추가
3. 규제 캘린더 변경을 사람이 수동 반영하는 대신, 최소한 "언제 업데이트가 필요한지" 알림이라도 자동화

## 7. 마무리

이 파이프라인을 정식으로 맡게 된다면, 지금은 월 단위 스냅샷으로 멈춰 있는 분석을 트렌드 비교까지 확장하고, CS팀 실측 데이터로 임팩트 수치를 검증된 값으로 바꿔나가고 싶다.

내가 이 문제를 비교적 빠르게 풀 수 있었던 이유는, 이전에 무신사 리뷰 데이터를 통해 고객 VoC를 유형별로 분석해본 경험이 있었기 때문이다. "고객의 말을 몇 가지 기준으로 나누는 것"과 "그 기준이 흔들리는 경계 케이스를 어떻게 처리할지 원칙을 세우는 것"이 VoC 분석의 진짜 난이도라는 걸 그 경험에서 미리 알고 있었고, 그래서 이번에도 분류 기준과 경계 케이스 처리 원칙을 문서로 먼저 고정하는 데 시간을 많이 썼다.

## (별첨) 기술 & 논리 상세

### 분류 체계
- **4유형**: 불만 / 기능 요청 / 칭찬 / 일반 문의
- **15소분류**: 유형별 3~4개 (예: 불만 → 시스템오류/계산오류/기능제한/연동오류)
- **5원인태그**: 규제불안 / 제품결함 / 사용법미숙 / 기한임박 / 원인불명
- 판단 원칙: "고객의 주된 기대 행동"이 무엇인가로 불만/기능요청 경계를 나눔(기존 것을 고쳐달라=불만, 없는 것을 새로 만들어달라=기능요청). 상세 경계 케이스와 근거는 `decisions.md` D-04~D-10 참조.

### 우선순위 점수식 (제품 개선 기획안 선정)
```
점수 = 빈도 점수(최대 5) + 임팩트 점수(최대 9)
임팩트 점수 = OEM 고객 포함(+3) + URGENT 포함(+3) + 제품결함 태그(+2) + 규제불안 태그(+1)
HIGH(8점↑) / MEDIUM(5~7점) / LOW(~4점)
```
