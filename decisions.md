## [Step 3] 분류 기준 설계

### D-04 4유형 정의 기준

| 유형 | 판단 기준 | 핵심 신호 |
|------|-----------|-----------|
| **불만** | 현재 플랫폼·서비스의 오류·불편·실망을 표현 | "안됩니다", "틀린 것 같습니다", "불편합니다", "반려됐습니다" |
| **기능 요청** | 새로운 기능 추가 또는 기존 기능 개선을 명시적으로 요청 | "기능을 추가해주세요", "지원해줄 수 있나요", "있으면 좋겠습니다" |
| **칭찬** | 플랫폼·서비스·담당자에 대한 긍정 경험 표현 | "감사합니다", "만족스럽습니다", "덕분에", "인상적이었습니다" |
| **일반 문의** | 규제·절차·방법론에 대한 정보성 질문 (플랫폼 자체 문제 아님) | "언제인가요", "어떻게 되나요", "있나요", "맞나요" |

#### Few-shot 참조 예시

**불만 (Complaint)**
> "지난주에 Scope 3 배출량 데이터를 입력하고 저장했는데 오늘 확인해보니 절반이 사라져 있어요. 5월 31일 CBAM 제출 기한이 코앞인데 이게 말이 됩니까? 빨리 복구해주세요."
> "PCF 자동계산 결과가 저희가 수동으로 계산한 값이랑 15% 이상 차이납니다. 이 데이터로 OEM에 보고했다가 나중에 문제 생기면 누가 책임지나요? 계산 로직 좀 제대로 검토해주세요."
> "온보딩 때 담당자분이 '2차 협력사 데이터 연동 쉽다'고 했는데, 실제로 해보니 협력사마다 포맷이 달라서 수작업으로 다 맞춰야 하더라고요. 완전히 낚인 기분이에요."

**기능 요청 (Feature Request)**
> "현재 리포트가 PDF로만 나오는데, 저희 내부 ERP에 붙이려면 엑셀(xlsx) 형식으로도 내보낼 수 있어야 합니다. CBAM 신고 양식에 맞게 컬럼 매핑까지 되면 더 좋겠어요."
> "협력사별 데이터 제출 현황을 한눈에 볼 수 있는 대시보드가 있으면 좋겠어요. 지금은 어느 협력사가 제출했고 어느 곳이 안 됐는지 일일이 확인해야 해서 관리가 너무 힘들어요."
> "CSRD 보고서 작성 시 ESRS 기준 항목별로 어떤 데이터를 채워야 하는지 체크리스트 형태로 가이드해주는 기능이 있으면 좋겠습니다. 저희 팀이 처음이라 뭘 어떻게 준비해야 할지 막막해요."

**칭찬 (Praise)**
> "솔직히 처음엔 반신반의했는데, 카본링크 도입 후 Scope 3 데이터 수집 시간이 3주에서 이틀로 줄었어요. 현대차 쪽 제출 기한 맞추는 게 이렇게 편해질 줄 몰랐습니다. 잘 쓰고 있어요."
> "CBAM 전환기 동안 규제 내용이 계속 바뀌어서 저희도 혼란스러웠는데, 담당 매니저분이 변경사항 생길 때마다 먼저 연락 주시고 리포트 양식도 바로 업데이트해주셨어요. 덕분에 신고 기한 문제없이 통과했습니다. 감사해요."
> "타사 솔루션도 검토해봤는데 카본링크가 국내 공급망 구조에 맞게 설계돼 있어서 저희 같은 2차 협력사가 쓰기에 훨씬 실용적이더라고요. 주변 협력사에도 추천하고 있습니다."

**일반 문의 (General Inquiry)**
> "저희 회사가 EU에 알루미늄 부품을 수출하는데, CBAM 적용 대상인지 확인하고 싶어요. 연간 수출량이 크지 않은 경우도 신고해야 하나요?"
> "카본링크에서 생성된 PCF 리포트가 CSRD 검증 시 제3자 인증 기관에서 그대로 인정받을 수 있나요? 어떤 인증 기준을 따르는지 알고 싶습니다."
> "현재 무료 체험 중인데 본 계약 시 1차 협력사 10개사 + 2차 협력사 50개사 규모면 요금제가 어떻게 되나요? 견적 받을 수 있을까요?"

### D-05 경계 케이스 처리 기준

**케이스 A — 불만 + 기능요청 혼합 (id 008, 043)**
- id 008: "입력 기능이 너무 불편합니다 ... 시간이 너무 걸립니다" → **불만**
- id 043: "기능이 너무 제한적입니다 ... 기능 개선 요청드립니다" → **기능 요청**
- **원칙 (v2)**: 고객의 **주된 기대 행동**이 무엇인가로 판단.
  - "기존 기능을 고쳐달라 / 현재 상태가 잘못됐다" → 불만 (CS팀 라우팅)
  - "없는 기능을 새로 만들어달라 / 기능을 추가·개선해달라" → 기능 요청 (제품팀 라우팅)
  - ※ 구버전 "마지막 문장" 기준은 문장 순서에 따라 결과가 달라지는 불안정성이 있어 폐기.
  - ※ id 008: 기대 행동 = "지금 불편한 입력 방식을 개선해달라" → 기존 수정 요구 → 불만 유지.
  - ※ id 043: 기대 행동 = "고정 템플릿 제약을 없애는 새 기능 추가" → 신규 추가 요구 → 기능 요청 유지.

**케이스 B — 계산 오류 의심 (id 014, 033)**
- id 014: "카본링크 계산 결과와 외부 기관 값이 20% 차이, 근거를 알고 싶다" → **불만**
- id 033: "경쟁사 플랫폼과 30% 차이, 계산 로직 공개 요청" → **불만**
- **원칙**: 수치 차이 문제 제기는 플랫폼 신뢰 훼손이므로 불만으로 분류.
  - id 033의 "설명해주세요"는 새 기능 추가 요청이 아닌 신뢰 해소 요청 → 불만 유지.

**케이스 C — 긴급 외부 규제 문의 (id 045)**
- "CBAM 신고 마감일이 지났는데 EU 세관에 문의하는 방법을 모르겠습니다. 수출 계약이 위험합니다." → **일반 문의**
- **원칙**: 분류(유형)와 긴급도(대응 속도)는 별개의 차원. 플랫폼이 원인이 아니면 일반 문의가 맞음.
- **보완 — urgent 플래그 (v2 추가)**:
  - 분류와 별개로 `urgent=Y` 플래그를 부여. 아래 키워드 중 하나 이상 포함 시 자동 플래그:
    - `긴급` / `마감` / `마감일이 지났` / `수출 계약` / `반려` / `위험` / `즉시`
  - id 045: 일반문의 + **urgent=Y**
  - id 010: 불만 + **urgent=Y** (EU 세관 반려)
  - urgent=Y 건은 CS팀 당일 에스컬레이션 대상으로 별도 처리.

**케이스 D — 가능 여부 문의 (id 042)**
- "카본링크에서 SBTi 시나리오 분석이 가능한가요?" → **일반 문의**
- **원칙**: 기능 존재 여부를 모르는 상태에서 확인하는 의문형 = 일반 문의.
  - "가능한가요?" → 일반 문의
  - "가능하게 해주세요 / 기능을 추가해주세요" → 기능 요청

**케이스 E — 미정의 케이스 처리 원칙 (신규 추가)**
- 새 데이터에서 위 케이스 A~D로 귀속이 어려운 경계 케이스 발생 시:
  - **확신도 낮음 플래그**: Claude가 분류 시 2개 이상 유형이 동점이면 `review=Y` 플래그 부여
  - `review=Y` 건은 사람이 최종 확인 후 분류 확정
  - 확정된 판단 근거는 decisions.md에 케이스 F, G, H... 순서로 추가 → 다음 달 few-shot 참조 소스로 누적
  - **원칙**: 경계 케이스는 파이프라인이 틀리는 것이 아니라 **새 케이스를 학습하는 기회**로 정의.

### D-07 Cause Tag 정의

| Cause Tag | 의미 | 적용 조건 |
|-----------|------|-----------|
| 규제불안 | 규제 변화·기준 불확실성에서 비롯된 문의·요청 | 규제 기한·적용 범위·해석 혼란이 명시된 경우 |
| 제품결함 | 플랫폼 버그·계산 오류·연동 불안정 등 제품 품질 문제 | 시스템 동작 이상·수치 불일치가 명시된 경우 |
| 사용법미숙 | 플랫폼 기능 존재하나 사용 방법 미숙지 | 기능 있음이 확인되나 사용자가 모르는 경우 |
| 기한임박 | 제출 마감·계약 기한 압박이 요인 | 긴급·마감·계약 위험 표현이 있는 경우 |
| 원인불명 | 위 4개 태그 해당 없음 (일반적 니즈·개선 요청) | 불만·문의 원인이 특정 요인으로 귀속 불가한 경우 |

### D-08 소분류 경계 케이스

**케이스 E — id 010: 시스템오류 vs 계산오류**
- "양식 문제인지 수치 문제인지 확인 부탁"→ 원인 미확인 상태
- EU 세관 반려는 보고서 양식(포맷/템플릿) 불충족이 수치 오류보다 빈도가 높음
- **결정**: 시스템오류 (양식·출력 문제 1순위로 추정). 수치 오류로 판명 시 다음 분류 사이클에서 계산오류로 정정.

**케이스 F — id 027: 시스템오류 vs 기능제한**
- "데이터 내보내기가 안 됩니다 + 인증 기관 요청 양식과 호환 안 됨"
- 내보내기 기능은 있으나 특정 형식을 지원하지 않음 → 기능 범위 제한
- **결정**: 기능제한 (기능 부재가 아닌 형식 비지원)

**케이스 G — id 043: 기능제한(불만) vs 보고서자동화(기능요청) 소분류**
- 대분류는 D-05 케이스 A 원칙에 따라 **기능 요청** 확정
- 소분류: 보고서 커스터마이징 = 보고서자동화 범주
- **결정**: 기능요청-보고서자동화

### D-12 인사이트 도출 (3-Layer 적용)

#### 인사이트 1 — CBAM 첫 신고 시즌이 만든 구조적 문의 집중

**Layer 1 (변화 감지)**
- 일반 문의: 전반기 6건 → 후반기 9건 (+50%, +3건) — 임계값 초과
- CBAM절차 소분류: 일반 문의 15건 중 7건(47%) 단일 최다

**Layer 2 (규제 캘린더 × Cause Tag)**
- 2026-05-31 CBAM 마감이 수집 기간 내 포함 → 마감 임박 문의 집중 구조 설명
- 마감 이후에도 id 045(6/14) 발생 → 마감 후 대응 문의로 전환
- Cause Tag: CBAM절차 7건 중 6건이 규제불안, 1건이 기한임박
- CBAM 2026 전면 시행 첫 해 → 고객 대부분이 처음 경험하는 신고 절차

**Layer 3 (관찰/조건/맥락/행동)**

> **관찰**: 전체 VoC 44건 중 CBAM 관련 10건(22.7%). 일반 문의 15건의 47%가 CBAM절차 소분류에 집중. urgent=Y 2건(id 010, id 045) 모두 CBAM 관련.
>
> **조건**: id 010(5/11, EU 세관 보고서 반려)은 마감 20일 전 긴급 발생. id 045(6/14, 마감 초과·수출 계약 위험)는 마감 후 2주 시점. 두 건 모두 OEM 고객 — 가장 규모가 크고 CBAM 노출이 높은 세그먼트.
>
> **맥락**: CBAM 2026 전면 시행으로 처음 신고하는 고객이 대다수. 플랫폼이 보고서를 생성해줘도 절차·기한 안내가 없으면 고객이 마감을 놓치거나 잘못된 양식으로 신고하는 구조적 리스크가 존재. id 010의 EU 세관 반려는 동일 양식을 쓰는 다른 OEM 고객 전체로 확산 가능한 시스템 리스크.
>
> **행동**: (1) CBAM 신고 단계별 체크리스트를 플랫폼 내 구축 — 데이터 준비 → 양식 작성 → 제출 → 확인 4단계. (2) 마감 30일·7일 전 자동 알림 기능 우선 개발. (3) id 010 EU 세관 반려 양식 즉각 QA — 동일 양식 사용 OEM 고객 전체에 선제 공지.

#### 인사이트 2 — 규제 복잡도 증가가 만든 자동화 수요 급증

**Layer 1 (변화 감지)**
- 기능 요청: 전반기 4건 → 후반기 8건 (+100%, +4건) — 임계값 초과
- 전체 기능 요청 12건 중 보고서자동화·규제가이드 2개 소분류가 8건(67%) 독점

**Layer 2 (규제 캘린더 × Cause Tag)**
- CSRD 2026 대상 확대 진행 중 → 고객이 CBAM·CSRD·TCFD를 동시 대응해야 하는 상황
- 기능 요청 12건의 cause tag: 규제불안 7건(58%) + 원인불명 5건 — 규제 대응이 핵심 동인
- 2027 CSRD 중소기업 확대 예정 → 현재 요청들이 2년 후 수요를 선행하는 신호

**Layer 3 (관찰/조건/맥락/행동)**

> **관찰**: 기능 요청이 후반기에 2배 급증. 12건 중 8건(67%)이 보고서자동화·규제가이드 2개 소분류에만 집중. 규제 대응 자동화 수요가 VoC를 끌어당기는 단일 최대 동인.
>
> **조건**: id 022(TCFD+CSRD 이중 보고 중복 제거), id 037(분기별 자동 생성), id 035(탄소크레딧 vs 감축 구분 표시), id 043(커스터마이징) — 각기 다른 고객이 독립적으로 "보고 복잡도 증가"를 호소. 동일 증상이 4건 이상 독립 발생하면 개인 니즈가 아닌 시장 니즈로 해석.
>
> **맥락**: CSRD 2026 대상 확대로 CBAM+CSRD+TCFD를 동시에 대응해야 하는 고객이 증가. 카본링크가 "각 규제별 리포트 생성"은 가능해도 "규제 간 데이터 재활용·자동 매핑"은 아직 미지원. 수작업 보고는 규모가 커질수록 오류 위험이 높아져 플랫폼 신뢰도 직결.
>
> **행동**: (1) 다중 프레임워크(CBAM+CSRD+TCFD) 데이터 자동 매핑 기능을 다음 분기 로드맵 최우선 과제로 격상. (2) 보고서 커스터마이징 기능(id 043) 단기 스프린트 포함. (3) 2027 CSRD 중소기업 확대 대비 규제가이드 기능 선제 개발 착수.

#### 인사이트 3 — 계산 신뢰 위기: 3건의 독립적 수치 불일치 호소

**Layer 1 (변화 감지)**
- 불만 4→5건: 임계값 미달(+25%) — Layer 1에서는 신호 없음
- **그러나 내용 분석(Layer 2)에서 이상 신호 감지** → Layer 1 미감지 케이스의 중요성 예시

**Layer 2 (규제 캘린더 × Cause Tag)**
- 불만 9건 cause tag 전부 제품결함(100%) — CS 이슈가 아닌 개발팀 이슈
- 계산오류 3건(id 014, 019, 033): 각기 다른 고객·날짜·채널에서 독립 발생
- id 019: IPCC 6차 보고서 GWP 미반영 → 배출계수 DB 업데이트 누락 = 전체 고객에 동일 오류 적용 가능성
- 규제 맥락: CBAM 신고·CSRD 보고 모두 계산 정확도가 법적 책임과 직결

**Layer 3 (관찰/조건/맥락/행동)**

> **관찰**: 불만 9건의 cause tag가 100% 제품결함. 이 중 계산오류 3건(id 014: 외부기관 대비 20% 차이, id 019: IPCC 6차 GWP 미반영, id 033: 경쟁사 대비 30% 차이)이 서로 다른 고객으로부터 독립적으로 발생.
>
> **조건**: id 010(EU 세관 보고서 반려)까지 포함하면 계산 또는 양식 정확도 의문 제기 VoC가 4건. 3건의 계산오류는 모두 다른 고객유형(2차 협력사 2건, 기타 1건)·다른 채널·다른 날짜 → 특정 고객 문제가 아닌 플랫폼 품질 문제.
>
> **맥락**: CBAM·CSRD 보고 맥락에서 수치 오류는 단순 불편이 아니라 법적 책임 문제로 전환. 고객이 카본링크 데이터로 OEM 또는 EU에 보고했다가 감사에서 문제가 생기면 플랫폼 신뢰도 붕괴 + 계약 분쟁 리스크. id 019의 IPCC 6차 GWP 미반영은 시스템 전체 문제일 가능성이 높음.
>
> **행동**: (1) id 019 IPCC 6차 GWP 즉각 업데이트 + 영향받는 전 고객 데이터 재산정·공지(긴급). (2) 배출계수 DB 버전 관리 체계 도입 — 업데이트 시 변경 이력과 영향 범위 자동 공지. (3) "외부 기준값 대비 검증 리포트" 기능 개발 — 고객이 스스로 ISO·경쟁사·공공 DB와 비교 가능한 투명성 기능.

## [Step 6] Standard — 재사용 가능한 파이프라인 설계

### D-13 리포트 섹션 구성 근거 (CS팀·제품팀 관점)

CSV를 업로드하면 아래 6개 섹션이 자동 생성된다. 각 섹션의 존재 이유와 수신자를 명시한다.

| 섹션 | 주 수신자 | 설계 근거 |
|------|----------|-----------|
| ① 핵심 요약 (Executive Summary) | CS팀장·제품 PM | 전체를 읽기 전에 당장 행동이 필요한 사안(URGENT, 계산오류 스파이크)을 3줄로 전달. 리포트를 끝까지 못 읽는 의사결정자를 위한 섹션. |
| ② 유형별 통계 카드 | CS팀 전체 | 당일 대응 우선순위 판단 기준. URGENT 건수가 0이면 루틴 처리, 1 이상이면 에스컬레이션 프로세스 진입. |
| ③ 유형 분포 + 기능요청 소분류 차트 | 제품팀 PM | 로드맵 우선순위 입력 데이터. 기능요청 소분류 상위 항목이 다음 스프린트 후보. 단순 건수가 아닌 비율로 표시해 기간 길이가 다른 데이터 간 비교 가능하게 설계. |
| ④ 고객세그먼트 × VoC 유형 교차표 | CS팀 + 영업팀 | OEM/1차/2차 협력사별로 어떤 유형의 문제가 집중되는지 파악. 세그먼트별 CS 대응 전략과 영업 이탈 리스크 판단에 사용. |
| ⑤ 전체 분류표 (필터 포함) | CS 담당자 (개인) | 본인이 담당하는 고객·채널 건만 필터링해서 확인. URGENT 행 클릭 시 상세 내용·에스컬레이션 지침 노출. |
| ⑥ 자동 인사이트 3개 | CS팀장·제품 PM | 관찰-맥락-행동 구조로 수치 해석과 다음 행동을 묶어서 전달. 숫자를 읽는 시간을 줄이고 "무엇을 해야 하는가"에 집중하도록 설계. |

**섹션 순서의 원칙**: 긴급도 높은 것 → 집계 → 패턴 → 개별 건 → 해석. 위에서 아래로 내려갈수록 세부화되는 구조. CS팀장은 ①②만 읽어도 당일 운영 판단 가능, 담당자는 ⑤를 중심으로 사용.

### D-14 파이프라인 재현성 설계 원칙

**목표**: 새 VoC CSV를 업로드하면 사람의 개입 없이 동일 품질의 리포트가 생성되어야 한다.

**재현성을 보장하는 3가지 장치**

1. **분류 기준의 문서화 (decisions.md)**
   - 4유형·15소분류·5원인태그의 정의와 경계 케이스 판단 원칙이 이 문서에 고정.
   - 새 CSV에서 새 경계 케이스가 발생하면 decisions.md에 추가 → 다음 실행의 few-shot 기준으로 누적.
   - "이 분류가 왜 이렇게 됐는가"를 재현 가능한 근거로 남기는 것이 파이프라인의 핵심.

2. **keyword_hint 기반 자동 분류 우선, content 보조**
   - keyword_hint가 있으면 결정론적으로 분류 → 동일 입력에 동일 출력 보장.
   - keyword_hint 없는 경우만 content 키워드 매칭으로 fallback → 비결정성을 최소화.

3. **날짜 정규화 내장**
   - `YYYY/MM/DD`, `N월 M일` 등 3가지 입력 형식을 `YYYY-MM-DD`로 자동 변환.
   - 새 CSV에 혼합 형식이 들어와도 정제 단계에서 흡수 → 집계·정렬 오류 없음.

**재현성의 한계 (명시적 비포함 사항)**
- `keyword_hint`가 없고 content 키워드도 매칭 안 되는 경우 → `일반문의`로 기본 분류 (보수적 fallback)
- 새 소분류가 필요한 경우 → decisions.md 업데이트 + 코드 수정 필요 (완전 자동화 미지원)
- 분류 정확도 검증 → 파이프라인이 자동 제공하지 않음, 사람의 샘플 검토 권장

## [Step 7] Challenge — 제품 개선 기획안 자동 생성

### D-15 기획안 선정 기준 (v2 — 점수 기반으로 개정)

> **v1 → v2 개정 이력**: 초기에는 기능요청 건수 1위·불만 건수 1위를 각각 고정 선정했으나, 선정 기준(건수)과 채점 기준(임팩트 점수)이 따로 노는 구조적 문제를 발견해 v2로 개정했다. 상세 내용은 D-30 참고.

업로드된 CSV에서 자동으로 기획안 2개를 선정한다.

**v2 선정 방식 (방식 A — 점수 기반)**
- 불만·기능요청 소분류 전체를 대상으로 D-16 점수를 산정한다.
- 점수 내림차순으로 정렬해 상위 2개를 기획안 01·02로 선정한다.
- 유형(불만/기능요청) 구분 없이 점수만으로 순위를 매긴다.

**선정 대상**: 불만·기능요청 유형만 포함. 칭찬·일반 문의는 제품 개선 액션으로 이어지기 어려워 제외.

**선정 개수가 2개인 이유**: 하나만 선정하면 제품팀(기능 추가) 또는 CS팀(문제 해결) 중 한 쪽이 빠질 수 있음. 3개 이상이면 실행 우선순위가 분산됨. 2개가 "가장 급한 것 하나씩" 전달하는 최소 단위.

두 개를 선정하는 이유: 제품팀(기능 추가)과 CS팀(문제 해결)이 각각 다른 액션을 취해야 하는 대표 케이스를 하나씩 커버하기 위함.

### D-16 우선순위 점수 산정 기준

점수 = 빈도 점수 + 임팩트 점수

**빈도 점수 (최대 5점)**
- 해당 소분류 건수 / 전체 건수 × 10, 최대 5점으로 캡
- 근거: 빈도가 높아도 임팩트가 낮으면 과대평가되는 문제를 방지

**임팩트 점수 (최대 9점)**

| 조건 | 점수 | 근거 |
|------|------|------|
| OEM 고객 포함 | +3 | CBAM 직접 노출·계약 금액 최대 세그먼트. 이탈 시 매출 영향 가장 큼 |
| URGENT 건 포함 | +3 | 당일 에스컬레이션 대상 = 즉각 대응 없으면 계약 분쟁·법적 리스크 |
| cause tag 제품결함 | +2 | 플랫폼 버그·계산오류 = 해당 소분류 고객 전체에 동일 문제 잠재 |
| cause tag 규제불안 | +1 | 외부 규제 변화가 동인 = 플랫폼 대응 필요하나 긴급도는 낮음 |

**등급 기준**
- HIGH (8점↑): 즉시 스프린트 편입 검토
- MEDIUM (5~7점): 다음 분기 로드맵 후보
- LOW (~4점): 백로그 등록 후 재평가

### D-22 gspread v6 호환성 문제 해결

**문제**: `AttributeError: 'Client' object has no attribute 'auth'`

**원인**: gspread v6에서 `client.auth` 속성이 제거됨. v5까지는 `client.auth.service_account_email`로 서비스 계정 이메일을 조회할 수 있었으나 v6부터 불가.

**해결**:
- `get_client()` 함수가 `(client, creds)` 튜플을 반환하도록 변경.
- `client.auth.service_account_email` → `creds.service_account_email`로 대체.

```python
# 수정 전
def get_client():
    creds = Credentials.from_service_account_file(...)
    client = gspread.authorize(creds)
    return client  # creds 버림

# 수정 후
def get_client():
    creds = Credentials.from_service_account_file(...)
    client = gspread.authorize(creds)
    return client, creds  # creds 함께 반환
```

- 구글시트 집계요약 탭에도 소분류별 건수·비율이 있음.
- 전체 분류표에서 필터 버튼으로 유형별 필터링 시 소분류도 확인 가능.
- 중복 정보 제거 → 리포트 흐름이 URGENT → 통계 카드 → 차트 → 분류표 → 인사이트 → 기획안으로 단순화됨.

### D-29 VoC 내용 전문 표시 (truncation 제거)

**문제**: `generate_html()`의 테이블 행 생성 시 `r["content"][:55]`로 55자에서 잘라 `…`을 붙였음. 실제 고객 피드백을 전체 읽을 수 없는 구조.

**결정**: 잘라내기 로직 완전 제거. `r["content"]` 원문 전체를 그대로 출력.

```python
# 제거 전
short_content = r["content"][:55] + ("…" if len(r["content"]) > 55 else "")

# 제거 후
# short_content 변수 자체 삭제 → r["content"] 직접 사용
```

**URGENT 알림 섹션도 동일 처리**: `r["content"][:60]…` → `r["content"]` 전문.

**근거**:
- CS 담당자가 고객 피드백을 55자만 보고 대응 방향을 결정하는 것은 현실적이지 않음.
- 구글시트 분류결과 탭에서는 전문이 표시됨 → HTML과 불일치 해소.
- 테이블 컬럼에 `min-width: 280px` 적용으로 전문 표시 시 레이아웃 유지.

## [Step 10] 기획안 선정 방식 개선 — 방식 A 채택

### D-30 기존 건수 기반 선정의 문제점 발견

**문제**: D-15 v1의 "건수 1위 소분류 고정 선정"과 D-16의 "임팩트 점수 채점" 사이에 구조적 불일치가 있었다.

**구체적 시나리오**:
- 시스템오류(2건): OEM 고객 + URGENT 포함 → 임팩트 점수 HIGH
- 보고서자동화(4건): OEM 없음, URGENT 없음 → 임팩트 점수 LOW~MEDIUM

기존 방식에서는 건수가 많은 보고서자동화가 기획안 01로 선정되고, 실제로 더 급한 시스템오류는 기획안에 아예 포함되지 않음.

**결론**: 선정 기준과 채점 기준이 분리되어 있으면 "높은 점수인데 선정 안 됨" 또는 "낮은 점수인데 선정됨"의 역설이 발생. 사용자 입장에서 기획안의 우선순위 근거를 신뢰하기 어려움.

### D-31 방식 A 채택 및 구현

**결정**: 불만·기능요청 소분류 전체를 임팩트 점수로 줄 세워 상위 2개를 선정한다.

**구현 변경**:
```python
# 변경 전: 유형별 건수 1위 고정
feat_sub.most_common(1)[0]  # 기능요청 건수 1위
comp_sub.most_common(1)[0]  # 불만 건수 1위

# 변경 후: 점수 기준 상위 2개
ranked = sorted(
    sub_groups.items(),
    key=lambda item: score(item[1], total)[2],  # total_s 기준
    reverse=True,
)
top2 = ranked[:2]
```

**효과**:
- 선정 기준과 채점 기준이 동일 (점수 하나로 통일).
- OEM 고객 포함·URGENT 포함 소분류가 건수 열세에도 기획안에 올라올 수 있음.
- "이 기획안이 왜 1위인가"를 점수로 명확히 설명 가능.

**잠재적 한계**:
- 불만 소분류가 높은 점수를 독점하면 기능요청 기획안이 기획안 01·02에 모두 빠질 수 있음. 이 경우 CS팀은 커버되지만 제품팀 입력이 누락될 수 있음.
- 현재 데이터 기준으로는 발생하지 않으나, 향후 데이터에서 발생하면 D-15를 재검토해야 함.

### D-33 PowerShell Set-Location -LiteralPath 문제 발견

**문제**: Anaconda Prompt(PowerShell)에서 `cd "C:\...\[01] 리뉴어스랩"` 실행 시 "경로를 찾을 수 없습니다" 오류 발생.

**원인**: PowerShell에서 `[`와 `]`는 와일드카드 패턴 문자로 해석됨. `[01]`을 "0 또는 1에 해당하는 한 글자" 패턴으로 처리해 실제 폴더명과 불일치 발생.

**해결**:
```powershell
# 잘못된 방법 (오류 발생)
cd "C:\Users\Minsoo\.claude\skills\AX해커톤\[01] 리뉴어스랩"

# 올바른 방법 (LiteralPath로 와일드카드 해석 비활성화)
Set-Location -LiteralPath "C:\Users\Minsoo\.claude\skills\AX해커톤\[01] 리뉴어스랩"
```

**HOW_TO_USE.md 반영**: ② 터미널 열기 단계에 `Set-Location -LiteralPath` 명령어와 주의사항을 명시. `cd` 명령어 사용 금지를 굵게 표시.

## [Step 12] 규제 캘린더 외부화

### D-34 규제 맥락 텍스트를 코드에서 분리

**문제**: 인사이트 카드의 맥락 텍스트("CBAM·CSRD·TCFD 동시 대응으로...")가 `classify_voc.py` 코드 안에 하드코딩되어 있었음. 규제 환경이 바뀌면 Python 코드를 직접 수정해야 하는 구조.

**결정**: 규제 정보를 `context/reg-calendar.md` 마크다운 표로 분리. 코드는 이 파일을 읽어서 규제명을 동적으로 조합.

```
# 업데이트 방법 (Python 불필요)
AI에게: "CSDDD 마감 2027-03-01 추가해줘"
→ context/reg-calendar.md 표에 한 줄 추가
→ 다음 classify_voc.py 실행 시 인사이트 자동 반영
```

**reg-calendar.md 컬럼 구조**

| 컬럼 | 역할 |
|------|------|
| 규제명 | 인사이트 맥락 텍스트에 삽입되는 이름 |
| 시행시작 | 현재 활성 여부 판단 기준 |
| 주요마감 | 인사이트 1 마감 임박 문구에 자동 삽입 |
| 관련소분류 | 어떤 인사이트 카드에 이 규제명이 쓰일지 결정 |
| 설명 | 사람이 읽는 맥락 메모 (코드에서 미사용) |

**인사이트별 규제 매핑 원칙**
- 인사이트 1 (규제 문의 집중): CBAM절차·CSRD해석·기타규제 관련 규제 → 활성 규제 목록 + 마감일
- 인사이트 2 (자동화 수요): 보고서자동화·규제가이드 관련 규제 → 다중 프레임워크 나열
- 인사이트 3 (품질 위기): CBAM절차·CSRD해석 관련 규제 → "XX·YY 보고 맥락에서 법적 책임"

**한계**: 규제 변경을 파이프라인이 자동 감지하지 않음. 담당자가 변경을 인지하고 파일을 수동 업데이트해야 함. 규제는 사람이 결정하는 영역이므로 완전 자동화는 구조적으로 불가.

### D-35 URGENT 분류 기준과 한계

**결정**: `content` 또는 `keyword_hint`에 아래 키워드 중 하나 이상 포함 시 `urgent=Y` 플래그 부여.

```python
URGENT_KW = [
    "긴급합니다", "긴급 대응", "긴급 처리", "마감일이 지났",
    "수출 계약이 위험", "수출 계약", "반려됐습니다", "위험합니다",
    "즉시", "계약이 위험", "긴급",
]
```

**근거**: CS팀이 당일 에스컬레이션 여부를 판단할 때 "고객이 긴급하다고 명시했는가"가 가장 신뢰도 높은 신호.

**한계 (명시적 비포함 사항)**
- 고객이 차분한 말투로 써도 내용상 긴급한 경우 미감지. 예: "EU 세관에서 보고서가 반려됐는데 확인 부탁드립니다"(긴급 단어 없음) → urgent 미분류.
- 키워드 목록은 현재 수집된 VoC 패턴 기반. 새로운 긴급 표현이 등장하면 `URGENT_KW`에 수동 추가 필요.
- URGENT 분류(유형)와 긴급도(대응 속도)는 별개 차원. 일반 문의도 urgent=Y 가능 (id 045 사례).

**중요**: 이 수치는 실측값이 아니라 **추정치**임. CS팀 인터뷰나 타임로그를 확보하지 못했으므로, `data/voc.csv` 건수와 컨설팅 업계 VoC 처리 실무를 근거로 다음과 같이 역산함.

**Before (수작업) 산출 근거**
- 모수: `data/voc.csv` 기준 월평균 약 30건 (5~6월 45건 ÷ 1.5개월)
- 건당 소요 시간 가정:
  - 채널별(이메일/채널톡/미팅메모) 원본 확인·취합: 1.5분
  - 내용 읽고 4유형 판단(경계 케이스 재확인 포함): 2.5분
  - 엑셀에 분류결과·키워드 수기 입력: 1분
  - → 건당 5분 × 30건 = 150분
- 월말 일괄 작업: 유형별 건수·비율 수기 집계 20분 + 규제 캘린더 대조하며 인사이트 도출 40분 = 60분
- **합계: 150분 + 60분 = 210분(3.5시간)/월**

**After (자동화) 산출 근거**
- CSV 교체: 1분
- `/analyze` → `/generate` 파이프라인 실행 대기: 3분
- 생성된 리포트 검토·확인: 15분
- **합계: 19분 → 약 20분**

**왜 이 가정을 썼는가**: 건당 처리 단계(취합→판단→입력)를 실제 파이프라인 처리 단계(정제→분류→집계)와 1:1로 대응시켜, 자동화가 대체하는 단계만 시간 산정에 포함시킴. 과장 방지를 위해 "읽는 시간 자체"(콘텐츠 이해)는 After에도 남겨둠(리포트 검토 15분).

**리스크/주의사항**: 발표 자료·리포트에 이 수치를 쓸 때는 반드시 "추정치" 또는 "가정 기준"이라고 명시할 것. CS팀 실측 데이터를 확보하면 이 섹션을 갱신해야 함.

**후속 결정 (2026-07-02) — 포트폴리오에서 철회**: 위 산출 근거(건당 1.5분/2.5분/1분, 리포트 검토 15분 등)가 실측 없이 소수점까지 쪼개져 있어 오히려 "안 재봤다"는 인상을 강화하고, 정밀한 합계(210분→20분)가 근거의 약함을 정밀함으로 가리는 형태가 됐다고 판단. 포트폴리오 슬라이드(7·8페이지)에서 시간 추정 수치를 모두 제거하고, voc.csv에서 직접 산출되는 건수 지표(예: "44건 → 0건 수작업 입력", "2/44건 URGENT 자동 감지")로 대체함. 이 섹션은 산출 방식에 대한 기록으로만 남기고, 향후 CS팀 실측 타임로그를 확보하면 그때 다시 시간 지표를 되살릴 것.
